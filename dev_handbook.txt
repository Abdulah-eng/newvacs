Inside that file there is a <strong>React web application</strong> that was already compiled and made small (“minified”). This application is a <strong>training simulator for pharmacy students</strong>. The student opens a fake patient’s medical chart, “talks” to a simulated patient by typing questions, writes a clinical note, and the app gives the student a score. You will mostly change small pieces of this file. Because the code is minified (variable names are short like zf, qf, Bu), <strong>you must edit it with small, exact text replacements using a script, never by hand</strong>, and you must <strong>test after every change</strong>. The rest of this document explains all of this slowly.</p><p><a id="read-this-first-one-paragraph"></a></p><h2><a id="Xcc58ad7a1c2af7d99fdb8268ec7ccacc38c75f6"></a>1. Glossary A — Medical words (the developer does not need to be a clinician)</h2><p>You do <strong>not</strong> need to understand medicine to work on this code. But you will see these words in the data and on the screen. Here is a simple meaning for each one. Treat them as <strong>labels for data fields</strong>, not as something you must judge.</p><p><strong>About the setting</strong></p><ul><li><strong>Ambulatory care</strong> = medical care for a patient who visits a clinic and goes home the same day. The patient is <strong>not</strong> staying in a hospital. (“Outpatient” means the same thing.)</li><li><strong>Pharmacist</strong> = a medicine expert. In this app, the pharmacist (and the student playing the pharmacist) helps manage long-term diseases together with the patient’s doctor.</li><li><strong>Preceptor</strong> = the teacher who supervises the student.</li><li><strong>Standardized patient</strong> = a fake/simulated patient used for training. In this app the patient is software: it answers the student’s typed questions.</li><li><strong>Chronic disease</strong> = a long-term disease that does not go away (for example, diabetes).</li><li><strong>EMR (Electronic Medical Record)</strong> = the computer chart that holds a patient’s information. The whole app looks like an EMR.</li></ul><p><strong>The patient’s note format (very important — this is the structure of the app)</strong></p><ul><li><strong>SOAP note</strong> = the standard way clinicians write a note. It has four parts:<ul><li><strong>S = Subjective</strong> = what the patient <em>says</em> (their story, symptoms, history).</li><li><strong>O = Objective</strong> = <em>measured</em> facts (vital signs, laboratory results, the physical exam).</li><li><strong>A = Assessment</strong> = the clinician’s <em>analysis</em> — a list of the patient’s problems and the reasoning about each one.</li><li><strong>P = Plan</strong> = what to <em>do</em> — medicine changes, monitoring, counseling, follow-up.</li><li>Remember this: <strong>S → O → A → P</strong>. The whole app and the grading are organized around it.</li></ul></li></ul><p><strong>Common diseases in the data</strong></p><ul><li><strong>Hypertension (HTN)</strong> = high blood pressure.</li><li><strong>Blood pressure (BP)</strong> = written as two numbers like 146/92. Lower is usually better.</li><li><strong>Type 2 diabetes mellitus (T2DM)</strong> = high blood sugar disease.</li><li><strong>A1C (also HbA1c)</strong> = a blood test, written as a percent like 8.2%, that shows average blood sugar over about 3 months. Lower is usually better; a common goal is below 7%.</li><li><strong>Hyperlipidemia</strong> = high blood fats (cholesterol).</li><li><strong>LDL</strong> = “bad” cholesterol. <strong>HDL</strong> = “good” cholesterol. <strong>Triglycerides</strong> = another blood fat.</li><li><strong>Obesity</strong> = too much body weight. <strong>BMI</strong> = a number from weight and height; about 30+ is obesity.</li><li><strong>CKD (chronic kidney disease)</strong> / <strong>DKD (diabetic kidney disease)</strong> = long-term kidney damage.</li><li><strong>eGFR</strong> = a kidney function number. Higher is better; below 60 is a concern.</li><li><strong>UACR / albuminuria</strong> = a urine test for kidney damage. Higher is worse.</li><li><strong>Creatinine, potassium, sodium</strong> = blood tests.</li></ul><p><strong>Medicines you will see (these are just product names in the data)</strong></p><ul><li><strong>metformin, lisinopril, hydrochlorothiazide, atorvastatin, acetaminophen, ibuprofen</strong> = common medicines in the cases.</li><li><strong>statin</strong> = a cholesterol medicine class (atorvastatin is a statin).</li><li><strong>GLP-1 RA</strong> and <strong>SGLT2 inhibitor</strong> = newer diabetes/kidney medicine classes.</li><li>A medicine is described by <strong>drug, dose, route, frequency</strong> (example: <em>metformin XR, 1000 mg, by mouth, once daily</em>). “Route” = how it enters the body (for example, “by mouth”). “PRN” = “as needed”.</li></ul><p><strong>Other words</strong></p><ul><li><strong>PMH</strong> = Past Medical History (diseases the patient had before).</li><li><strong>HPI</strong> = History of Present Illness (the story of the current visit).</li><li><strong>ROS</strong> = Review of Systems (a checklist of symptoms the patient denies or confirms).</li><li><strong>Vitals / Vital signs</strong> = blood pressure, heart rate, temperature, etc.</li><li><strong>Labs</strong> = laboratory blood/urine test results.</li><li><strong>Allergies / NKDA</strong> = “NKDA” = “No Known Drug Allergies”.</li><li><strong>Immunizations / Vaccines</strong> = shots like influenza (flu), pneumococcal, zoster, Tdap.</li><li><strong>Adherence</strong> = whether the patient takes the medicine as told. “Nonadherence” = missing doses.</li><li><strong>ICD-10</strong> = a standard code for a diagnosis (example: I10).</li><li><strong>MRN</strong> = Medical Record Number. <strong>PCP</strong> = Primary Care Provider (the patient’s main doctor).</li><li><strong>Medication reconciliation</strong> = checking and confirming the patient’s true medicine list. <em>(Note: the interactive reconciliation feature was removed — see Section 13.)</em></li></ul><p><a id="Xcc58ad7a1c2af7d99fdb8268ec7ccacc38c75f6"></a></p><h2><a id="glossary-b-technical-words"></a>2. Glossary B — Technical words</h2><ul><li><strong>HTML file</strong> = the single file you are editing. You open it in a web browser to run the app.</li><li><strong>React</strong> = a JavaScript library for building user interfaces from “components”.</li><li><strong>Component</strong> = a function that returns a piece of screen. In this code, components are functions with short names like zf, qf, Bf.</li><li><strong>Props</strong> = the input values given to a component (like function arguments).</li><li><strong>State</strong> = data that can change while the app runs (for example, what the student typed).</li><li><strong>Hook</strong> = a React function for state, like useState. In this file React is stored in the variable _, so you will see _.useState, _.useRef, _.useEffect, _.useMemo, _.useCallback, and _.Fragment.</li><li><strong>Minified / bundle</strong> = the code was made small and hard to read for production. Variable names are short and there are almost no spaces or new lines. <strong>This is why you edit with exact text matching, not by hand.</strong></li><li><strong>JSX factory</strong> = instead of normal JSX (&lt;div&gt;), the compiled code calls functions:<ul><li>a.jsx(type, props) builds an element that has <strong>one</strong> child.</li><li>a.jsxs(type, props) builds an element whose children are a <strong>fixed array</strong>.</li><li>type is either a string like "div" or a component function like zf.</li></ul></li><li><strong>localStorage</strong> = small storage inside the web browser. The app saves the student’s work there.</li><li><strong>node --check</strong> = a command that checks if JavaScript has correct syntax (no broken code).</li><li><strong>Harness</strong> = a small test script we write to <em>run</em> a piece of the real code with fake inputs, to confirm it behaves correctly. (Syntax check is not enough; a harness catches logic errors.)</li><li><strong>Rubric</strong> = a grading sheet. Here, a Word document (.docx) that lists every fact a perfect note must contain, with points.</li><li><strong>Keyword matching</strong> = the grader checks if certain words/numbers appear in the student’s text.</li><li><strong>Point-weighting</strong> = each gradable item has a point value; the score is earned points divided by total points.</li></ul><p><a id="glossary-b-technical-words"></a></p><h2><a id="what-the-app-does-the-students-journey"></a>3. What the app does (the student’s journey)</h2><ol><li>The student picks a <strong>patient</strong> and a <strong>day</strong> (an “encounter”).</li><li>The app shows the patient’s <strong>chart</strong> (Snapshot, Vitals, Labs, Medications, Allergies, etc.). Some information is <strong>hidden on purpose</strong> — the chart is incomplete.</li><li>The student opens <strong>Patient Interview</strong> and <strong>types questions</strong> to the simulated patient. The patient answers using pre-written responses. When the student asks about a hidden topic, that topic becomes <strong>“discovered”</strong> and the student documents it.</li><li>The student writes the <strong>Assessment</strong> and builds the <strong>Plan</strong>.</li><li>The student opens <strong>SOAP Note</strong>, can press <strong>Generate Draft</strong> (the app fills a draft from the student’s inputs), edits it, then presses <strong>Grade Against Model SOAP</strong>.</li><li>The app compares the note to the <strong>rubric</strong> for that exact encounter and shows a <strong>score</strong> and a list of <strong>what was done well</strong> and <strong>what is missing</strong>.</li></ol><p>The teaching idea: <em>the student must interview the patient to uncover hidden facts, then document them.</em> Do not break this idea (see Section 14).</p><p><a id="what-the-app-does-the-students-journey"></a></p><h2><a id="the-patients-and-the-nine-encounters"></a>4. Their PROBLEMS (and some MEDICATIONS) are also <strong>references</strong> to the Tuesday arrays.</li><li><strong>Linda</strong> is the same pattern (Tuesday object is <strong>ji</strong>; Wed/Thu reference it).</li><li><strong>Consequence:</strong> if you change Tuesday’s PROBLEMS or MEDICATIONS, the change <strong>also appears</strong> on that patient’s Wednesday and Thursday automatically. This is usually what you want, but be aware of it.</li><li><strong>Gotcha:</strong> when you search for an array like PROBLEMS:, check the next character. Do not edit the wrong one.</li></ul><p><a id="Xb5f113f2bebfea878b719daaac6f1becbda4d11"></a></p><h2><a id="X1622404e1a5993f1eba09c524195950c3834885"></a>9. Engine 1 — The Patient Interview matcher (how the simulated patient answers)</h2><p>When the student types a question, this engine chooses the patient’s answer.</p><p><strong>Data per encounter:</strong> - <strong>INTERVIEW_FIELDS</strong> = the documentation boxes shown to the student: [{ key, label, placeholder }]. - <strong>INTERVIEW_KNOWLEDGE</strong> = the patient’s brain: [{ id, topic, field, keywords, response }]. - keywords = words that, if found in the student’s question, choose this answer. - response = the patient’s exact reply text. - field = if present, asking this marks that field as <strong>discovered</strong>.</p><p><strong>The matching functions:</strong> - <strong>Vf(text)</strong> = <em>normalize</em>: make lowercase, replace anything that is not a letter/number with a space, collapse spaces, and put a space at the start and end. - <strong>$f(normalizedText, keywords)</strong> = <em>score</em>: for each keyword found, add 1 + min(words−1, 2) (longer keyword phrases score higher). - <strong>Qf(question, knowledgeArray)</strong> = <em>choose the answer</em>. The order of checks is: 1. If the question is empty → a friendly prompt. 2. <strong>__CONV</strong> (conversational layer) — if the student made a <em>statement</em> (not a question), for example praise, empathy, transition, or asking permission, return a short neutral reply (like “Okay.” / “Thank you.”) that adds <strong>no</strong> new medical facts. 3. <strong>Clinical match</strong> — score every INTERVIEW_KNOWLEDGE item; the highest score wins (ties → the earlier item). If a winner exists, return its response. 4. If no clinical match: <strong>Kf</strong> = greeting words → a greeting reply. 5. <strong>__DFLT</strong> = standard default answers for common topics that this patient has no specific answer for (for example “no recreational drugs”). 6. <strong>ud</strong> = a generic “I am not sure what you mean” fallback. - <strong>Yf</strong> = a small hash used to pick a varied response from a list.</p><p><strong>Two screens use the same documentation record:</strong> - The <strong>Subjective</strong> tab (qf) and the <strong>Quick documentation</strong> panel inside <strong>Patient Interview</strong> (Jf) show the <strong>same fields</strong> and save to the <strong>same record</strong>. They must stay <strong>identical (1:1)</strong>. If you change one, change the other the same way.</p><p><a id="X1622404e1a5993f1eba09c524195950c3834885"></a></p><h2><a id="X5c8fdf2209383968ad7a110245dafbe770d5538"></a>10. Engine 2 — The SOAP grading (the rubrics are the source of truth)</h2><p>This is the most important recent feature. The <strong>9 Word rubric files are the answer keys.</strong></p><p><strong>The rubric files</strong> (one per encounter): Week_1_Rubric_Pt_A_Tuesday.docx … Week_1_Rubric_Pt_C_Thursday.docx (Pt A = Maria, Pt B = James, Pt C = Linda). Each rubric lists <strong>200+ discrete inputs</strong>. Each input has an id (like 1B.5), a description, a <strong>point value</strong>, and full/half/zero credit rules. The five rubric sections are: 1 = Subjective, 2 = Objective, 3 = Assessment, 4 = Plan, 5 = Presentation Quality. All points add up to <strong>100</strong>.</p><p><strong>How the rubrics are stored in the code — the answer key Bu:</strong> - <strong>Bu</strong> is one big object: Bu[caseId] = { subjective:[…], objective:[…], assessment:[…],   plan:[…], avoid:[…] }. - Each gradable item is { id, label, pts, keywords }: - id = the rubric input id (for example "1B5"). - label = the rubric input text (shown to the student as feedback). - pts = the point value from the rubric. - keywords = normalized words/values to search for in the student’s note (for example "146 92" for a blood pressure, "2000 mg", "metformin"). - Rubric <strong>Sections 1–4</strong> become the four Bu sections. Rubric <strong>Section 5 (Presentation)</strong> is about structure/formatting, which keyword matching cannot judge fairly, so it is <strong>not</strong> part of the automatic score. So the score denominator is the rubric’s content points (about 92–95). - avoid = “watch-out” items (things the student should <strong>not</strong> do). Maria has 4 each; James and Linda have an empty avoid array.</p><p><strong>The grader functions:</strong> - <strong>pd(text)</strong> = normalize text the same way as the interview (lowercase, punctuation → space, collapse spaces, pad). <strong>Important:</strong> because punctuation becomes a space, 146/92 becomes 146 92. That is why keywords are stored in this spaced form. - <strong>Ha(text, keywords)</strong> = returns true if <strong>any</strong> keyword is found inside the text. - <strong>sg(answerKey, note)</strong> = the grader.
