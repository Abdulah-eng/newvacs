Read this first (one paragraph) VACS-Demo.html  is  one single HTML file . Inside that file there is a  React web application  that was already compiled and made small (“minified”). This application is a  training simulator for pharmacy students . The student opens a fake patient’s medical chart, “talks” to a simulated patient by typing questions, writes a clinical note, and the app gives the student a score. You will mostly change small pieces of this file. Because the code is minified (variable names are short like  zf ,  qf ,  Bu ),  you must edit it with small, exact text replacements using a script, never by hand , and you must  test after every change . The rest of this document explains all of this slowly. 1. Glossary A — Medical words (the developer does not need to be a clinician) You do  not  need to understand medicine to work on this code. But you will see these words in the data and on the screen. Here is a simple meaning for each one. Treat them as  labels for data fields , not as something you must judge. About the setting Ambulatory care  = medical care for a patient who visits a clinic and goes home the same day. The patient is  not  staying in a hospital. (“Outpatient” means the same thing.) Pharmacist  = a medicine expert. In this app, the pharmacist (and the student playing the pharmacist) helps manage long-term diseases together with the patient’s doctor. Preceptor  = the teacher who supervises the student. Standardized patient  = a fake/simulated patient used for training. In this app the patient is software: it answers the student’s typed questions. Chronic disease  = a long-term disease that does not go away (for example, diabetes). EMR (Electronic Medical Record)  = the computer chart that holds a patient’s information. The whole app looks like an EMR. The patient’s note format (very important — this is the structure of the app) SOAP note  = the standard way clinicians write a note. It has four parts: S = Subjective  = what the patient  says  (their story, symptoms, history). O = Objective  =  measured  facts (vital signs, laboratory results, the physical exam). A = Assessment  = the clinician’s  analysis  — a list of the patient’s problems and the reasoning about each one. P = Plan  = what to  do  — medicine changes, monitoring, counseling, follow-up. Remember this:  S → O → A → P . The whole app and the grading are organized around it. Common diseases in the data Hypertension (HTN)  = high blood pressure. Blood pressure (BP)  = written as two numbers like  146/92 . Lower is usually better. Type 2 diabetes mellitus (T2DM)  = high blood sugar disease. A1C (also HbA1c)  = a blood test, written as a percent like  8.2% , that shows average blood sugar over about 3 months. Lower is usually better; a common goal is below  7% . Hyperlipidemia  = high blood fats (cholesterol). LDL  = “bad” cholesterol.  HDL  = “good” cholesterol.  Triglycerides  = another blood fat. Obesity  = too much body weight.  BMI  = a number from weight and height; about  30+  is obesity. CKD (chronic kidney disease)  /  DKD (diabetic kidney disease)  = long-term kidney damage. eGFR  = a kidney function number. Higher is better; below  60  is a concern. UACR / albuminuria  = a urine test for kidney damage. Higher is worse. Creatinine, potassium, sodium  = blood tests. Medicines you will see (these are just product names in the data) metformin, lisinopril, hydrochlorothiazide, atorvastatin, acetaminophen, ibuprofen  = common medicines in the cases. statin  = a cholesterol medicine class (atorvastatin is a statin). GLP-1 RA  and  SGLT2 inhibitor  = newer diabetes/kidney medicine classes. A medicine is described by  drug, dose, route, frequency  (example:  metformin XR, 1000 mg, by mouth, once daily ). “Route” = how it enters the body (for example, “by mouth”). “PRN” = “as needed”. Other words PMH  = Past Medical History (diseases the patient had before). HPI  = History of Present Illness (the story of the current visit). ROS  = Review of Systems (a checklist of symptoms the patient denies or confirms). Vitals / Vital signs  = blood pressure, heart rate, temperature, etc. Labs  = laboratory blood/urine test results. Allergies / NKDA  = “NKDA” = “No Known Drug Allergies”. Immunizations / Vaccines  = shots like influenza (flu), pneumococcal, zoster, Tdap. Adherence  = whether the patient takes the medicine as told. “Nonadherence” = missing doses. ICD-10  = a standard code for a diagnosis (example:  I10 ). MRN  = Medical Record Number.  PCP  = Primary Care Provider (the patient’s main doctor). Medication reconciliation  = checking and confirming the patient’s true medicine list.  (Note: the interactive reconciliation feature was removed — see Section 13.) 2. Glossary B — Technical words HTML file  = the single file you are editing. You open it in a web browser to run the app. React  = a JavaScript library for building user interfaces from “components”. Component  = a function that returns a piece of screen. In this code, components are functions with short names like  zf ,  qf ,  Bf . Props  = the input values given to a component (like function arguments). State  = data that can change while the app runs (for example, what the student typed). Hook  = a React function for state, like  useState . In this file React is stored in the variable  _ , so you will see  _.useState ,  _.useRef ,  _.useEffect ,  _.useMemo ,  _.useCallback , and  _.Fragment . Minified / bundle  = the code was made small and hard to read for production. Variable names are short and there are almost no spaces or new lines.  This is why you edit with exact text matching, not by hand. JSX factory  = instead of normal JSX ( &lt;div&gt; ), the compiled code calls functions: a.jsx(type, props)  builds an element that has  one  child. a.jsxs(type, props)  builds an element whose children are a  fixed array . type  is either a string like  &quot;div&quot;  or a component function like  zf . localStorage  = small storage inside the web browser. The app saves the student’s work there. node --check  = a command that checks if JavaScript has correct syntax (no broken code). Harness  = a small test script we write to  run  a piece of the real code with fake inputs, to confirm it behaves correctly. (Syntax check is not enough; a harness catches logic errors.) Rubric  = a grading sheet. Here, a Word document ( .docx ) that lists every fact a perfect note must contain, with points. Keyword matching  = the grader checks if certain words/numbers appear in the student’s text. Point-weighting  = each gradable item has a point value; the score is earned points divided by total points. 3. What the app does (the student’s journey) The student picks a  patient  and a  day  (an “encounter”). The app shows the patient’s  chart  (Snapshot, Vitals, Labs, Medications, Allergies, etc.). Some information is  hidden on purpose  — the chart is incomplete. The student opens  Patient Interview  and  types questions  to the simulated patient. The patient answers using pre-written responses. When the student asks about a hidden topic, that topic becomes  “discovered”  and the student documents it. The student writes the  Assessment  and builds the  Plan . The student opens  SOAP Note , can press  Generate Draft  (the app fills a draft from the student’s inputs), edits it, then presses  Grade Against Model SOAP . The app compares the note to the  rubric  for that exact encounter and shows a  score  and a list of  what was done well  and  what is missing . The teaching idea:  the student must interview the patient to uncover hidden facts, then document them.  Do not break this idea (see Section 14). 4. Engine 1 — The Patient Interview matcher (how the simulated patient answers) When the student types a question, this engine chooses the patient’s answer. Data per encounter:  -  INTERVIEW_FIELDS  = the documentation boxes shown to the student:  [{ key, label, placeholder }] . -  INTERVIEW_KNOWLEDGE  = the patient’s brain:  [{ id, topic, field, keywords, response }] . -  keywords  = words that, if found in the student’s question, choose this answer. -  response  = the patient’s exact reply text. -  field  = if present, asking this marks that field as  discovered . The matching functions:  -  Vf(text)  =  normalize : make lowercase, replace anything that is not a letter/number with a space, collapse spaces, and put a space at the start and end. -  $f(normalizedText, keywords)  =  score : for each keyword found, add  1 + min(words−1, 2)  (longer keyword phrases score higher). -  Qf(question, knowledgeArray)  =  choose the answer . The order of checks is: 1. If the question is empty → a friendly prompt. 2.  __CONV  (conversational layer) — if the student made a  statement  (not a question), for example praise, empathy, transition, or asking permission, return a short neutral reply (like “Okay.” / “Thank you.”) that adds  no  new medical facts. 3.  Clinical match  — score every  INTERVIEW_KNOWLEDGE  item; the highest score wins (ties → the earlier item). If a winner exists, return its  response . 4. If no clinical match:  Kf  = greeting words → a greeting reply. 5.  __DFLT  = standard default answers for common topics that this patient has no specific answer for (for example “no recreational drugs”). 6.  ud  = a generic “I am not sure what you mean” fallback. -  Yf  = a small hash used to pick a varied response from a list. Two screens use the same documentation record:  - The  Subjective  tab ( qf ) and the  Quick documentation  panel inside  Patient Interview  ( Jf ) show the  same fields  and save to the  same record . They must stay  identical (1:1) . If you change one, change the other the same way. 10. Engine 2 — The SOAP grading (the rubrics are the source of truth) This is the most important recent feature. The  9 Word rubric files are the answer keys. The rubric files  (one per encounter):  Week_1_Rubric_Pt_A_Tuesday.docx  …  Week_1_Rubric_Pt_C_Thursday.docx  ( Pt A  = Maria,  Pt B  = James,  Pt C  = Linda). Each rubric lists  200+ discrete inputs . Each input has an id (like  1B.5 ), a description, a  point value , and full/half/zero credit rules. The five rubric sections are: 1 = Subjective, 2 = Objective, 3 = Assessment, 4 = Plan, 5 = Presentation Quality. All points add up to  100 . How the rubrics are stored in the code — the answer key  Bu :  -  Bu  is one big object:  Bu[caseId] = { subjective:[…], objective:[…], assessment:[…],   plan:[…], avoid:[…] } . - Each gradable item is  { id, label, pts, keywords } : -  id  = the rubric input id (for example  &quot;1B5&quot; ). -  label  = the rubric input text (shown to the student as feedback). -  pts  = the point value from the rubric. -  keywords  = normalized words/values to search for in the student’s note (for example  &quot;146 92&quot;  for a blood pressure,  &quot;2000 mg&quot; ,  &quot;metformin&quot; ). - Rubric  Sections 1–4  become the four  Bu  sections. Rubric  Section 5 (Presentation)  is about structure/formatting, which keyword matching cannot judge fairly, so it is  not  part of the automatic score. So the score denominator is the rubric’s content points (about 92–95). -  avoid  = “watch-out” items (things the student should  not  do). Maria has 4 each; James and Linda have an empty  avoid  array. The grader functions:  -  pd(text)  = normalize text the same way as the interview (lowercase, punctuation → space, collapse spaces, pad).  Important:  because punctuation becomes a space,  146/92  becomes  146 92 . That is why keywords are stored in this spaced form. -  Ha(text, keywords)  = returns true if  any  keyword is found inside the text. -  sg(answerKey, note)  = the grader. It is  point-weighted : - For each section, for each item, if a keyword is found (in that section’s text  or  anywhere in the note), the item’s  pts  are earned. -  pct = round(earnedPoints / possiblePoints × 100) . - It also returns  earned / possible   counts  (number of items) for the per-section display, and the list of  hits  and  misses  per section, and the  flags  (any  avoid  items found). -  dg(pct)  = turns the percent into a label (“Strong”, “Solid”, “Developing”, “Needs work”). -  ug  = the SOAP Note screen. Buttons:  Generate Draft  (calls  ag , which builds a draft from the student’s inputs),  Grade Against Model SOAP  (calls  sg(Bu[caseId], note) ), and  Copy note . -  pg  = the feedback display (score circle, watch-outs, per-section “did well” / “to improve”). What “source of truth” means here:  a perfect note that contains every rubric fact scores  100% . This was tested for all 9 encounters. If you change a rubric, you must rebuild the matching  Bu[caseId]  from it (see Section 15.4). 11. Conversational rules for the simulated patient (do not break these) The patient’s neutral conversational replies ( __CONV ) must  never add new medical information . They only acknowledge the student (for example “Okay.”, “Thank you.”, “Of course.”). The patient must never invent new symptoms, history, medicines, habits, family history, or goals. Only real facts from the source documents (or the standard  __DFLT  defaults) may be revealed. Keep this rule if you touch the interview engine. 14. Rules you must NEVER break Never hand-type changes into the minified bundle.  Use the safe script method (Section 15). Never change the clinical facts.  The patient responses and chart data come from the author’s source documents. You may change how things look, not what the facts are. Never change navigation, colors, or unrelated sections  unless the task explicitly says so. Never delete teaching features:  the “intentionally missing” banner, the “✓ discovered” tags, auto-save, and “return to the interview” prompts. Never make the simulated patient add new medical information  in conversational replies. Never let the Subjective tab and the Quick documentation panel become different.  They must stay identical (same fields, same saved record). Always validate  with  node --check   and  a runtime harness before delivering (Section 16). Be careful with child-safety, medical-safety, and privacy  — but in this project you only touch a fixed training data set; do not add real patient data. 15. The safe way to make a change (step by step) 15.1 Files and locations Read-only original (do not edit):   /mnt/project/VACS-Demo__2_.html . Final output to deliver:  copy the finished file to  /mnt/user-data/outputs/VACS-Demo.html . 15.2 The edit method — exact, guarded text replacement (use Python) Because the code is minified, you find a unique exact string and replace it. Always assert that the string appears  exactly once , so you never edit the wrong place by accident. P  =   &apos;VACS-Demo.html&apos; data  =   open (P, encoding = &apos;utf-8&apos; ).read() old  =   &apos;...exact text from the file...&apos; new  =   &apos;...your replacement...&apos; assert  data.count(old)  ==   1 ,  f&quot;expected 1 match, got  { data . count(old) } &quot; data  =  data.replace(old, new) open (P,  &apos;w&apos; , encoding = &apos;utf-8&apos; ).write(data) If the same pattern appears in several encounters, do a  span-local  replace: first find the byte range of the one case you want (between its case id and the next), then replace only inside that range. This stops you from changing other encounters by mistake. 15.3 Unicode escaping gotcha (this causes silent failures) Some functions store special characters as  literal escape text  (the 6 characters  \u2014 ), and others store the  real character  ( — ). Examples: -  qf  (Subjective) often uses the  real characters . -  Jf ,  Uf ,  Bf  often use  \u2014 -style escapes  (a backslash, then  u , then digits). So when you write the  old  string in Python: - To match a  real  em dash, write  &quot;\u2014&quot;  (Python turns it into the real character). - To match the  literal escape  text in the file, write  &quot;\\u2014&quot;  (a real backslash +  u2014 ). If your  assert  says “0 matches”, this is usually the reason. Check the raw bytes first:  print(repr(data[i-20:i+20])) . 15.4 If you change a rubric (rebuild the grading key) Read the rubric  .docx  with:  extract-text file.docx  (this prints the rubric as markdown). Re-run the parser that turns each rubric input into a  {id, label, pts, keywords}  item, grouping rubric Sections 1–4 into subjective/objective/assessment/plan (skip Section 5). Rebuild  Bu[caseId]  and replace it in the file (keep the  avoid  array). Validate that a synthetic “perfect note” (all item labels) scores about  100% , an empty note is flagged empty, and a thin note scores low. 16. Part 3 — NAPLEX Quiz (the knowledge-check gate between visits) 3.1 What “NAPLEX” means (plain English) NAPLEX  =  North American Pharmacist Licensure Examination . It is the real exam a pharmacy student must pass to become a licensed pharmacist in the United States. In this app, the  NAPLEX Quiz  is a multiple-choice knowledge test written in the same style as that exam (“NAPLEX-inspired”). Its job is to check that the student learned the week’s clinical rules  before  they are allowed to see the patients. 3.2 Where the quiz fits in the week Week 1 is a five-day module: Monday  — Guideline Review  + NAPLEX Quiz  (first study, then test). Tuesday / Wednesday / Thursday  — three clinic  visits  with the simulated patients. Friday  — journal club. The quiz is a  gate . The student must reach the passing score before the next step unlocks. 3.3 The gating rule (the most important part) The quiz shows  10 questions at a time  (“populates 10 at a time”). The student must score  90% or higher  — that is,  9 of 10 correct  — to pass. Passing  unlocks the next visit . Design intent (confirmed):  the week’s full  40-question  bank is  split into four 10-question quizzes , and each quiz must be passed at  &gt;= 90%  for the student to  move to the next visit . Already built in the current code:  an immediate  per-question rationale  appears after each answer;  retries are unlimited ; and the  questions and the answer options reshuffle  on every attempt. The student’s  best score  is saved and shown in the Preceptor View. 3.4 The question bank = the uploaded file The uploaded file  Week_1_NAPLEX_Quiz_Questions.docx  is the  full Week 1 question bank . The whole file is the question  pool ; the app takes  10 questions at a time  from this pool to build each quiz. The bank is 40 questions. 20 hypertension  questions ( HTN-001  …  HTN-020 ) 20 type 2 diabetes  questions ( T2DM-001  …  T2DM-020 ) Three formats:  Single-best-answer  (22),  Select-all-that-apply  (12),  K-type-style  (6). The file is internally consistent: its title and summary both read  40 questions (20 + 20) , and the content matches. Instructions for the developer: Load all  40  questions as the week’s pool.
